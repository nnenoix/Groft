## Архитектурные конвенции

- `from __future__ import annotations`, PEP 604 unions, type hints везде.
- Без `print`/`logging` в best-effort операциях — ошибки глушатся silent.
- `asyncio.create_subprocess_exec` с list-args, никогда `shell=True`.
- Файловые операции в `run_in_executor`.

## core/paths.py

- `install_root()` (MEIPASS / repo root) и `user_data_root()` (`CLAUDEORCH_USER_DATA` env → fallback). Оба `@functools.cache`.
- `*_dir()` авто-mkdir. `config_path`/`default_config_path` без mkdir. Нет module-level констант путей.
- Тест: autouse-фикстура `_reset_path_cache` чистит `@cache` до/после каждого теста.

## ProcessBackend абстракция

- Protocol (`core/process/backend.py`): `spawn/send_text/kill/capture_output/is_alive/list_targets`.
- `TmuxBackend`: `send_text` split по `\n`, каждая строка → `["-l", "--", line]` + bare `Enter` — injection-guard.
- `WindowsBackend`: Popen-based; CRLF normalize; kill double-tap `terminate→wait(5)→kill`+`taskkill /T /F`. `CREATE_NEW_CONSOLE = getattr(subprocess, "CREATE_NEW_CONSOLE", 0)` — import-safe на Linux.
- `select_backend(config)` читает `config["process"]["backend"]`; `auto` → POSIX→Tmux, Windows→Windows.
- `InMemoryBackend` (`tests/support/`) для unit-тестов.
- Checkpoint-миграция: `tmux_target` → `target` при чтении старых DB.

## communication/server.py

- UI-forward триггер: `sender == "ui"` из WS registration.
- `UI_SINK_AGENT = "ui"`, `SNAPSHOT_SINK_AGENT = "opus"`.
- `type=decision` frame → forward к opus (сервер файл не пишет).

## core/messengers/webhook.py (Phase 6.3)

- `WebhookBridge(url, secret, template)` — stateless outbound notifier. Ctor валидирует: хотя бы один из `{event}/{agent}/{text}` в шаблоне, + после подстановки sentinel (`__probe_event__` etc.) должен парситься как JSON. `ValueError` иначе.
- **`str.replace`, не `str.format`** — иначе user content с литеральными `{}` ломает. Три токена: `{event}`, `{agent}`, `{text}`.
- `notify(event, agent, text)` → `httpx.Response`. Per-call `httpx.AsyncClient(timeout=10.0)`, headers `Content-Type: application/json` + `X-Webhook-Secret: <secret>`. Body через `content=` (bytes), не `json=` — шаблон уже валидный JSON.
- `test()` → bool; swallows network errors. Canned: event=`test`, agent=`groft`, text=`"Webhook test from Groft"`.

## communication/server.py — webhook REST (Phase 6.3)

- `_webhook_state_path()` → `.claudeorch/messenger-webhook.json`. `_read_webhook_state/_write_webhook_state` best-effort.
- `POST /messenger/webhook/configure {url, secret, template}`: shape-validate → 400; URL scheme http/https + netloc (`urlparse`) → 400 иначе; template делегируется `WebhookBridge(...)` ctor, `ValueError` → 400. Persist через executor; persist-exc → 500.
- `POST /messenger/webhook/test` — читает state, construct bridge, `await bridge.notify("test","groft",...)`. Response `{ok, status, error}`. 400 если config не существует или шаблон стал невалидным. Network error → 200 с `{ok:false, status:null, error:"ClassName: msg"}`.
- `GET /messenger/webhook/status` → `{status: "connected"|"not-connected", url}`. Нет `connecting` — стейтлесс.
- Bridge НЕ бутится на старте; construct per-call.

## core/messengers/discord.py (Phase 6.4)

- `DiscordBridge(token, orchestrator, *, allowlist, backend, state_path, client_factory, sync_guild_id)`. Token regex `^[A-Za-z0-9_-]{10,}\.[A-Za-z0-9_-]{5,}\.[A-Za-z0-9_-]{20,}$` (мирит с UI `DISCORD_TOKEN_RE`).
- Lazy import `discord`/`app_commands` в `_default_gateway` — отсутствие dep → warning, bridge idle-parks.
- `_default_gateway` создаёт `discord.Client(intents=Intents.default())` + `CommandTree`. На `on_ready` делает `tree.sync()` (или `sync(guild=...)` если задан `sync_guild_id`) и записывает `str(client.user)` в `_bot_user`.
- `stop()` awaits `client.close()` (coroutine!) потом cancel gateway task. `close` через `getattr`+`iscoroutine` — защищено от тестовых non-awaitable fakes.
- `_on_pair(interaction, code)` — ephemeral reply "Paired ✓"/"Invalid/expired code"/"Usage: /pair <code>".
- `_on_ask(interaction, agent, text)` — paired-user-only. Странгеры → `response.defer(ephemeral=True)` (нельзя не ответить на interaction, но ephemeral defer не раскрывает bot existence). Echo `[agent] sent: <first-line>` / `[agent] send failed`.
- `accept_pairing` — идентичная семантика Telegram-а (TTL 300s, single-use, persist в state file).

## communication/server.py — discord REST (Phase 6.4)

- `/messenger/discord/configure` `{token}` — format-only (regex `_DISCORD_TOKEN_RE`), persist `{token, bot_user: null|preserved, paired_user_id?}`. **НЕТ live probe** — discord.py требует gateway connect, слишком тяжело для REST.
- `/messenger/discord/start-pairing` — 6-char code из того же алфавита что Telegram, но store `_discord_pairs` (отдельный! не cross-leak с telegram). `_discord_clock` monkeypatchable.
- `/messenger/discord/status` → `{status, bot_user, paired_user_id}`. Та же tri-state логика: token+paired→connected, token-only→connecting, иначе not-connected.
- Constants: `DISCORD_PAIR_TTL=300.0`, `DISCORD_PAIR_CODE_LEN=6`.

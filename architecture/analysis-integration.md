## Notes

- Canonical WS schema is scattered across `server.py`, `client.py`, and `useOrchestrator.ts` docstrings; no single source of truth. A `communication/protocol.md` (or typed schema shared via codegen) would prevent drift â€” current divergences (`status` payload, `snapshot` agent field) are a direct consequence.
- The orchestrator's role on the WS is asymmetric: it publishes via `comm_client` but never consumes. That violates the symmetry the UI implicitly assumes (every `message` addressed to opus is a real request). Either consume or stop addressing opus.
- UI assumes the server forwards status frames it receives (`useOrchestrator.ts:110-115` comment admits the gap). `server.py:_dispatch` status branch was changed to forward to UI (`server.py:260-263`), so that comment is now stale â€” update the doc to reflect that only UI receives, not peers.
- `CommunicationClient` has no reconnect logic (`client.py:17-22`); MCP server uses it (`mcp_server.py:21-37`) and will be permanently dead if the orchestrator restarts. Not strictly a UIâ†”BE issue but invalidates any "agent stayed connected" assumption the UI banner suggests.

# Bugs & races (analysis-bugs)

## Critical
- [ ] `core/spawner.py:31-46` — `subprocess.run` called synchronously inside `async def spawn`, blocks the event loop while tmux starts; returncode is never checked so a failed `tmux new-window`/`send-keys` silently registers a phantom agent. — fix: switch to `asyncio.create_subprocess_exec`, await, verify `returncode == 0` before populating `self.active_agents` / invoking the register callback.
- [ ] `ui/src/hooks/useOrchestrator.ts:178-196` + `ui/src/store/agentStore.tsx:168-180` — snapshot handler appends the **entire** tmux capture-pane to `terminalOutput` on every change; `APPEND_TERMINAL` has no dedup, so each tick re-appends the full recent history until the 100-line buffer trims — terminal view shows duplicated lines. — fix: replace snapshot append with a replace-style action (`SET_TERMINAL { name, lines }`) that stores the snapshot as-is.
- [ ] `ui/src/hooks/useChannels.ts:85-92, 107-111` — user-supplied `token` / pairing `code` are interpolated into shell command strings (`/telegram:configure ${token}`, `/telegram:access pair ${code}`) and forwarded to `run_tmux_command`; no validation/escaping → arbitrary tmux keystroke injection. — fix: validate input against an allowlist regex and/or pass as a separate argv field into the Tauri command instead of string interpolation.
- [ ] `core/spawner.py:14` — `open(config_path)` at construction without context manager guard against `FileNotFoundError`/`YAMLError`; crashes the orchestrator on boot if config.yml is missing or malformed. — fix: wrap read in try/except and fall back to an empty models dict (same pattern as `main._load_tmux_config`).
- [ ] `communication/server.py:89-95` — uvicorn readiness loop polls `_uvicorn_server.started` for 5 s without ever checking `_uvicorn_task.done()`; if uvicorn crashes during bind (port in use, etc.) `start()` returns "ready" anyway and the REST endpoint is silently absent. — fix: break the poll when the task is done; re-raise the task exception.

## Important
- [ ] `communication/server.py:218` / `server.py:280-282` / `agent_watchdog.py:121` / `communication/mcp_server.py:35` — `loop.create_task(...)` / `asyncio.create_task(...)` results are not stored; per CPython docs these tasks may be garbage-collected before completion. — fix: keep a `self._background_tasks: set[asyncio.Task]` and `add_done_callback(self._background_tasks.discard)`.
- [ ] `core/watchdog/agent_watchdog.py:45-47` — `register_agent` unconditionally overwrites any existing `AgentState`, resetting `last_change_time`/`wake_fired`/`restart_fired`; if spawner re-registers an already-watched agent (restore path in `main.py:106-107` after a hot-reload), the stuck timers reset and a truly stuck agent can never fire `restart`. — fix: short-circuit when `name` already present, or merge fields instead of replacing.
- [ ] `core/session/checkpoint.py:85-94` — `save()` does `assert self._conn is not None` then awaits the lock and uses `self._conn` inside; concurrent `close()` can null `self._conn` between assert and use. — fix: capture `conn = self._conn` inside the lock, raise if None.
- [ ] `communication/server.py:186-188` — `_unregister` (sync) is called from the async `_handle_connection.finally` and from `_route_direct`/`_forward_to_ui` error paths. Inside, `asyncio.get_running_loop()` raises `RuntimeError` if ever invoked from a non-loop thread and is swallowed by `return` — roster goes stale on disconnect. — fix: make `_unregister` async, or always schedule via the captured loop reference stored at start().
- [ ] `communication/mcp_server.py:22-41` — module-level `inbox: list[dict]` mutated by `_consume()` and `get_messages()` without any lock; the `.copy()+clear()` in `get_messages` is non-atomic under asyncio task switches that may occur between `listen()` iterations. — fix: guard mutations with an `asyncio.Lock`, or swap the list with a single reference.
- [ ] `core/guard/process_guard.py:91-106` — a second SIGINT while `_confirming=True` is dropped; if the prompt hangs (no tty), the user has no escape besides SIGKILL. — fix: second signal should set `_shutdown_event` unconditionally.
- [ ] `core/main.py:65-111` — `comm_server.start()` is awaited **before** `process_guard.install()`; a SIGINT during startup propagates as `KeyboardInterrupt` and leaves the WS server / duckdb connection dangling. — fix: install signal handlers first (they only need a loop), then start subsystems.
- [ ] `ui/src/components/primitives/Composer.tsx:55-57` — `setTimeout(() => setThinking(false), 1800)` runs with no cleanup; if the component unmounts during those 1.8 s the setState fires on an unmounted component (React dev-mode warning, minor leak). — fix: store the timer id in a ref and clear it in an effect cleanup.
- [ ] `ui/src/components/AgentDrawer.tsx:9-45` — `Select` uses `defaultValue` and `Toggle` stores `checked` only in initial state, so prop changes (new agent selected) don't update either control. — fix: treat them as controlled components driven by props.
- [ ] `ui/src/hooks/useWebSocket.ts:155` — the effect depends on `[connect, clearReconnectTimer]`; `connect` is recreated whenever `scheduleReconnect` changes, which depends on `reconnectDelayMs` — swapping the delay tears down and re-opens the socket mid-session. — fix: keep `connect` stable via a ref, or depend only on `[url, agentName]`.

## Desirable
- [ ] `communication/client.py:33,46,54,66,79` — every send method does `assert self._ws is not None`; once disconnected, assertions raise `AssertionError` instead of a descriptive `RuntimeError`. — fix: raise a named exception so callers can differentiate.
- [ ] `communication/server.py:303-334` — `_forward_to_tmux` iterates lines and sends per-line; a tmux failure halfway through prints no diagnostic and leaves a half-typed command at the target. — fix: log the failure and abort the whole forward in one go.
- [ ] `core/recovery/recovery_manager.py:92-101` — missing tmux targets end up silently skipped from watchdog registration; noted only in duckdb `details`, never surfaced to the operator. — fix: add a stderr warning (goes together with task #17).
- [ ] `ui/src/App.tsx:80-85`, `ui/src/components/ComposerDock.tsx:12-15` — `setTimeout` focus calls never cleared; harmless on root components but leak timers during hot reloads.
- [ ] `ui/src/components/ComposerModal.tsx:13-16` — `requestAnimationFrame → setMounted` can resolve after unmount if cancel loses the race; use an `if (mounted)` guard inside the callback.
- [ ] `core/watchdog/agent_watchdog.py:90-104` — `_monitor_loop` never logs when `_capture` raises; truly-dead tmux sessions appear "active" forever. — fix: increment a miss counter per agent and escalate after N misses.

## Noted but out of scope
- DuckDB connection is reused across the default thread-pool executor via `run_in_executor`; serialized by `asyncio.Lock`, so in practice this is safe, but it is fragile to future parallelization. Not currently buggy.
- `core/main.py:140` `restart_claude_code` is a TODO stub wired to watchdog/error_handler callbacks; not a bug, but recovery is effectively inert until implemented.
- `communication/server.py:303` `_forward_to_tmux` echoes raw UI content to a terminal; acceptable since UI is trusted, but worth revisiting if untrusted clients ever connect.
- Module-level state in `mcp_server.py` (`client`, `_connected`, `inbox`) makes the module un-reimport-safe; fine for the single-process MCP lifecycle.

# ClaudeOrch — Система оркестрации агентов

## Твоя роль
Ты тимлид. Не исполнитель. Твоя задача:
- Получить задачу от пользователя
- Оценить сложность задачи и выбрать режим (Solo/Team)
- В Solo режиме — сделать самому через субагента
- В Team режиме — разбить на ТЗ и отдать исполнителям
- Следить за их работой через watchdog
- Проверить результат (код + тесты)
- Дать коррекции если нужно
- Перейти к следующим задачам

## Цикл работы — Гибридный режим

### Анализ задачи (всегда первый шаг)
Перед началом оцени сложность:
- Простая (1 модуль, < 2 часов работы) → Solo режим
- Сложная (несколько модулей, параллельная работа) → Team режим

### Solo режим (простые задачи)
1. Сам пишешь код через субагента
2. После каждого изменения → тесты
3. После каждого изменения → аудит кода
4. Проблема найдена → фикс → тесты → аудит снова
5. Всё чисто → коммит → следующая задача

### Team режим (сложные задачи)
1. Построй граф зависимостей → architecture/graph.md
2. Напиши короткое ТЗ на 1-2 задачи → architecture/current-task.md
3. Спауни нужных агентов через AgentSpawner (каждый в своём tmux окне)
4. После каждого цикла агентов → тесты → аудит
5. Проблема → коррекции агенту → тесты → аудит снова
6. 3 попытки не помогли → rollback → новый подход
7. Всё чисто → merge + коммит → следующие задачи

### TDD цикл (обязательно в обоих режимах)
Перед выдачей задачи исполнителю:
1. Сам напиши тест для задачи → architecture/current-test.md
2. Передай исполнителю задачу + тест
3. Исполнитель пишет код пока тест не проходит
4. Tester запускает тест → результат тебе
5. Прошёл → merge + коммит
6. Не прошёл → исполнитель правит → снова

### Постоянный аудит (в обоих режимах)
После каждого изменения кода проверяй:
- Архитектура: соответствует ли общему дизайну?
- Безопасность: нет ли уязвимостей?
- Паттерны: используются ли правильные подходы?
- Стыковка: работает ли с остальным кодом?
- Тесты: все ли случаи покрыты?

Фиксируй каждое решение в architecture/decisions.md.
Если аудит нашёл проблему — не переходи к следующей задаче.

## Критерии выбора режима
Solo если:
- Задача в одном файле или модуле
- Нет параллельных независимых частей
- Можно сделать последовательно без потери времени

Team если:
- Бэк и фронт можно писать одновременно
- Несколько независимых модулей
- Задача явно выиграет от параллельности

## Правила
- ТЗ должно быть коротким и конкретным — не более 20 строк
- В Team режиме — не пиши код сам, только через исполнителей
- В Solo режиме — пиши через субагента, не напрямую
- После каждого решения записывай в architecture/decisions.md
- Всегда проверяй результат перед переходом к следующей задаче
- Если агент не отвечает — watchdog сообщит тебе
- Тесты и аудит обязательны после каждого изменения в обоих режимах
- Не переходи к следующей задаче если аудит нашёл проблему

### Атомарность задач
Задача готова к выдаче только если:
- Описывается одним предложением
- Затрагивает один файл или одну функцию
- Имеет чёткий критерий готовности
Если задача шире — разбей её на подзадачи.

### Diff ревью
При проверке результата читай только git diff:
git diff main..feature/{task_name}
Не читай весь файл — только изменения.
Это быстрее и точнее.

### Явные интерфейсы
Перед спауном исполнителей запиши в architecture/interfaces.md:
- Типы и интерфейсы (TypeScript/Python)
- Сигнатуры функций
- Формат входных и выходных данных
Исполнитель работает строго по интерфейсам.

### Контекст кэширование
CLAUDE.md и memory файлы передавай всегда первыми в контексте.
Они кэшируются автоматически — экономит время на каждой задаче.
Не меняй их структуру без необходимости.

## Исполнители
- backend-dev (Sonnet 4.6) — API, БД, бизнес логика
- frontend-dev (Sonnet 4.6) — React, UI, компоненты
- tester (Haiku 4.5) — тесты после каждого цикла
- reviewer (Sonnet 4.6) — финальный ревью

Исполнители запускаются как отдельные claude процессы в tmux окнах через
AgentSpawner. Opus координирует через WebSocket сервер и файловую систему.
Каждый исполнитель имеет свой AGENT_NAME и WS соединение.

## Структура проекта
- architecture/ — ТЗ, решения, граф зависимостей
- memory/ — память каждого агента
- tasks/ — backlog, current, done
- .claudeorch/ — системные данные оркестратора

## Конфигурация
Модели агентов задаются в config.yml в корне проекта.
При старте читай config.yml и используй указанные модели
при спауне исполнителей через AgentSpawner.

## Коммуникационный сервер
Все агенты общаются через WebSocket сервер на localhost:8765.
Когда тебе нужно передать сообщение исполнителю или получить
snapshot терминала — используй коммуникационный сервер.
REST API на localhost:8766/agents показывает кто подключён.

## Интеграция с Claude Design

ClaudeOrch умеет принимать handoff из Claude Design.

### Как это работает:
1. Пользователь создаёт дизайн в Claude Design
2. Экспортирует в Claude Code (кнопка handoff)
3. Оркестратор обнаруживает входящий дизайн пакет
4. Opus анализирует дизайн файлы и компоненты
5. Автоматически создаёт план реализации
6. Спаунит агентов для написания кода по дизайну

### Что приходит из Claude Design:
- HTML/CSS компоненты
- Дизайн система (цвета, типографика, компоненты)
- Структура страниц и экранов
- Описание интерактивности

### Задача Opus при получении handoff:
- Прочитать все файлы дизайна
- Составить план реализации компонентов
- Определить порядок — что делать первым
- Записать в architecture/design-handoff.md
- Начать цикл разработки
